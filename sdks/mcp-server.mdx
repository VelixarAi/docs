The Velixar MCP server exposes your memory API as [Model Context Protocol](https://modelcontextprotocol.io) tools — letting Claude Desktop, Kiro, Cursor, and other MCP clients store and search memories natively.

## Install

```bash
npm install -g velixar-mcp-server
```

## Configure

You only need your API key. Get one from [Settings → API Keys](https://velixarai.com/settings) in the Velixar dashboard.

Add to your MCP client config:

<CodeGroup>
```json Kiro (~/.kiro/settings/mcp.json)
{
  "mcpServers": {
    "velixar": {
      "command": "velixar-mcp-server",
      "env": {
        "VELIXAR_API_KEY": "vlx_your_key"
      }
    }
  }
}
```

```json Claude Desktop
{
  "mcpServers": {
    "velixar": {
      "command": "velixar-mcp-server",
      "env": {
        "VELIXAR_API_KEY": "vlx_your_key"
      }
    }
  }
}
```

```json Cursor
{
  "mcpServers": {
    "velixar": {
      "command": "velixar-mcp-server",
      "env": {
        "VELIXAR_API_KEY": "vlx_your_key"
      }
    }
  }
}
```
</CodeGroup>

That's it — restart your MCP client and the Velixar tools will be available.

## Auto-recall

When the server starts, it automatically fetches your most recent memories and exposes them as an MCP resource (`velixar://memories/recent`). Clients that support resources will see this context immediately — no manual search needed.

Set `VELIXAR_AUTO_RECALL` to `false` to disable, or adjust `VELIXAR_RECALL_LIMIT` to control how many memories are loaded.

## Tools

The server registers 5 tools:

| Tool | Description |
|---|---|
| `velixar_store` | Store a memory with content, tags, tier, and type |
| `velixar_search` | Semantic search across memories |
| `velixar_list` | List memories with pagination (returns full metadata including IDs) |
| `velixar_update` | Update an existing memory by ID |
| `velixar_delete` | Delete a memory by ID |

<Warning>
**Subagent Limitation**: Velixar MCP tools are only available in the **primary agent context**. If your AI assistant supports subagents (like Claude's `use_subagent`), those subagents will NOT have access to Velixar memory tools. Always handle memory operations in the primary agent before/after delegating other tasks to subagents.
</Warning>

### Workflow Patterns

<Tabs>
  <Tab title="✅ Correct">
    ```
    Primary Agent:
    1. velixar_search for context
    2. Delegate file operations to subagent
    3. velixar_store results
    ```
    
    **Example**: Primary agent searches memory → passes context to subagent for file creation → stores outcome in memory.
  </Tab>
  
  <Tab title="❌ Incorrect">
    ```
    Primary Agent:
    1. Delegate "search memory and create files" to subagent
    
    Subagent:
    - velixar_search → FAILS (tool not available)
    ```
    
    **Problem**: Subagents don't inherit MCP tools from the primary agent.
  </Tab>
</Tabs>

## Environment variables

| Variable | Required | Description |
|---|---|---|
| `VELIXAR_API_KEY` | Yes | Your API key from [Settings → API Keys](https://velixarai.com/settings). Starts with `vlx_`. |
| `VELIXAR_USER_ID` | No | An arbitrary label to tag memories (e.g. `kiro`, `claude`). Useful if you use multiple MCP clients and want to filter memories by source. Default: `mcp-user`. |
| `VELIXAR_API_URL` | No | Custom API base URL. Default: `https://api.velixarai.com` |
| `VELIXAR_AUTO_RECALL` | No | Fetch recent memories on startup. Default: `true` |
| `VELIXAR_RECALL_LIMIT` | No | Number of memories to auto-recall. Default: `10` |
