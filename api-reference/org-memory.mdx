## Overview

The Org Memory API enables hierarchical knowledge sharing across teams and organizations. Memories flow through scoped tiers with automatic vector embedding and audit logging.

**Scope flow:** `Personal → Workspace → Team → Org`

All endpoints (except `/embed`) require a valid Supabase JWT in the `Authorization: Bearer <token>` header.

**Base URL:** `https://api.velixarai.com/v1/org/memory`

---

## Commit Memory

Share a memory with your team workspace. Automatically generates a vector embedding and stores it in Qdrant.

<ParamField body="content" type="string" required>
  The memory content to commit (this gets embedded for semantic search)
</ParamField>

<ParamField body="workspace_id" type="string">
  Your org_workspace ID. Defaults to your membership's workspace.
</ParamField>

<ParamField body="team_id" type="string">
  Your team ID. Defaults to your membership's team.
</ParamField>

<ParamField body="tags" type="string[]">
  Optional tags for categorization
</ParamField>

<ParamField body="source_memory_id" type="string">
  Optional — the personal memory ID this was promoted from
</ParamField>

```bash
curl -X POST https://api.velixarai.com/v1/org/memory/commit \
  -H "Authorization: Bearer $SUPABASE_JWT" \
  -H "Content-Type: application/json" \
  -d '{
    "content": "Our API rate limit is 1000 req/min for Pro users",
    "team_id": "team_xyz",
    "tags": ["api", "rate-limits"]
  }'
```

**Response:** The created `org_memory` record with `scope: "workspace"` and a `qdrant_point_id`.

<Note>
  If `workspace_id` or `team_id` are omitted, they're resolved from your membership record.
</Note>

## Submit for Promotion

Request that a team memory be promoted to org-wide visibility. Creates an entry in the promotion queue for manager/admin review.

<ParamField body="memory_id" type="string" required>
  The memory to submit for promotion
</ParamField>

<ParamField body="team_id" type="string" required>
  The team the memory belongs to
</ParamField>

```bash
curl -X POST https://api.velixarai.com/v1/org/memory/submit \
  -H "Authorization: Bearer $SUPABASE_JWT" \
  -H "Content-Type: application/json" \
  -d '{"memory_id": "mem_abc", "team_id": "team_xyz"}'
```

## Promote to Org

<Warning>Requires `manager` or `admin` role.</Warning>

Promotes a team memory to organization scope. The memory is re-embedded into the org-level Qdrant collection, making it searchable by all teams.

<ParamField body="memory_id" type="string" required>
  The memory to promote
</ParamField>

<ParamField body="org_id" type="string" required>
  The organization ID
</ParamField>

```bash
curl -X POST https://api.velixarai.com/v1/org/memory/promote \
  -H "Authorization: Bearer $SUPABASE_JWT" \
  -H "Content-Type: application/json" \
  -d '{"memory_id": "mem_abc", "org_id": "org_xyz"}'
```

## Retrieve Memories

Semantic vector search across all scopes you have access to. Queries all your team and org collections in parallel, then merges and ranks by cosine similarity.

<ParamField body="query" type="string">
  Natural language search query. If omitted, returns recent memories.
</ParamField>

<ParamField body="limit" type="integer" default="20">
  Max results to return
</ParamField>

```bash
curl -X POST https://api.velixarai.com/v1/org/memory/retrieve \
  -H "Authorization: Bearer $SUPABASE_JWT" \
  -H "Content-Type: application/json" \
  -d '{"query": "rate limiting policy", "limit": 10}'
```

**Response:**
```json
{
  "memories": [
    {
      "id": "mem_abc",
      "content": "Our API rate limit is 1000 req/min for Pro users",
      "scope": "org",
      "relevance_score": 0.92,
      "tags": ["api", "rate-limits"],
      "committed_by": "user_123",
      "access_count": 5,
      "qdrant_point_id": "235e7a30-...",
      "created_at": "2026-02-21T..."
    }
  ],
  "scopes_queried": 3
}
```

<Info>
  The retrieve endpoint automatically searches all scopes your membership grants access to: personal, team(s), and org. Results are deduplicated and ranked by relevance score.
</Info>

## Embedding Pipeline

Memory embedding happens automatically via a Postgres trigger → Edge Function → Lambda pipeline:

<Steps>
  <Step title="Insert">
    A memory is inserted into `org_memories` (via commit, promote, or direct insert).
  </Step>
  <Step title="Trigger">
    Postgres trigger `trg_embed_org_memory` fires and calls the `embed-org-memory` edge function.
  </Step>
  <Step title="Edge Function">
    The edge function calls the Lambda `/embed` endpoint with a shared secret (`X-Embed-Key` header).
  </Step>
  <Step title="Lambda">
    The Lambda validates the secret, verifies the memory exists, generates an OpenAI `text-embedding-3-small` vector (1536 dimensions), and stores it in Qdrant.
  </Step>
  <Step title="Update">
    The `qdrant_point_id` is written back to the `org_memories` row, confirming the embedding is searchable.
  </Step>
</Steps>

The `/embed` endpoint is not user-facing — it uses a shared secret instead of JWT auth.

## Audit Log

All org memory operations are automatically logged to the `org_audit_log` table via database triggers.

**Auto-logged actions:**
| Action | Trigger |
|--------|---------|
| `memory.commit` | Memory inserted into `org_memories` |
| `memory.delete` | Memory deleted from `org_memories` |
| `promotion.approve` / `promotion.reject` | Promotion queue status change |
| `member.invite` / `member.remove` | Manual logging from dashboard |

Each audit entry includes: `actor_id`, `org_id`, `team_id`, `target_type`, `target_id`, and a `metadata` JSONB field with context (scope, content preview, email, role).

<Warning>
  The audit log tab in the dashboard is only visible to managers and admins.
</Warning>

## Scope Hierarchy

| Scope | Visibility | Who can write | Qdrant Collection |
|-------|-----------|---------------|-------------------|
| `personal` | Only you | Any authenticated user | `personal_{user_id}` |
| `workspace` | Your team | Team members | `team_{team_id}` |
| `org` | All teams in org | Promoted by manager/admin | `org_{org_id}` |

## Security

The org memory system uses a 6-layer security model:

1. **API Gateway JWT Authorizer** — Supabase JWT validated on all `/org/memory/*` routes (except `/embed`)
2. **Lambda Membership Check** — User must have an active membership to access any org data
3. **Role-Based Access Control** — Promote/queue/audit restricted to manager/admin roles
4. **Row-Level Security** — Supabase RLS policies ensure users only see their own org's data
5. **Embed Secret** — The `/embed` endpoint validates a shared secret from Secrets Manager
6. **Memory Existence Check** — The embed Lambda verifies the `memory_id` exists before creating vectors
